# WK — 에이전트 공통 안내 (모든 agent 필독)

이 파일은 이 리포에서 작업하는 **모든 에이전트**가 읽는 공통 규칙입니다.

## 블로그 모바일 우선(mobile-native) 원칙 — 페이지·글·표는 휴대폰에서 먼저 검증한다 (모든 agent 공통, 반드시 준수) — CMPA-255 보드 확정 2026-06-09

> **블로그(CaskCode)의 페이지·글·표는 "데스크톱이 아니라 휴대폰(모바일)에서 보기 좋게"
> 만드는 것을 기본 전제로 한다.** 독자 대다수가 모바일이고, 보드가 실제로 모바일에서
> 깨짐을 지적했다. 표·카드·이미지 등 어떤 산출물을 만들든 **좁은 화면(≈360px)에서
> 글이 잘리거나 가려지지 않는지** 먼저 확인하고 만든다.

- **(표 설계 — 컬럼 수를 줄여라)** 좁은 화면에서 컬럼이 많으면 글자가 뭉개지거나 잘린다.
  **데이터 표는 가능하면 2컬럼 `(주체, 상세)` 구조로 짠다.** 예: 가격변동 패치 표는
  `(위스키, 상세)` — `상세`에 면세가·국내최저·절약·할인율을 `<br>` 로 줄바꿈해 보기 좋게
  묶는다. 5~6컬럼을 모바일에 그대로 욱여넣지 않는다.
- **(가로 스크롤 폴백 — 글을 가리지 말 것)** 그래도 넓어지는 표(예: 본편 비교표)는
  **가로 스크롤이 가능해야 한다.** CSS에서 `.post table { display:block; overflow-x:auto; }`
  같은 가로 스크롤 컨테이너를 보장하라. **글자를 숨기거나(`overflow:hidden`) 잘라서
  안 보이게 만드는 것은 금지** — 보드 지적의 핵심이다("글이 안 보이면 안 된다").
- **(긴 텍스트 칸 — 한 칸에 몰지 말 것)** 위스키명처럼 긴 텍스트가 한 컬럼에 과도하게
  몰리면 모바일에서 줄이 무너진다. 텍스트를 `상세` 칸으로 분산하거나 컬럼을 합쳐
  **한 칸이 화면 폭을 독식하지 않게** 한다.
- **(검증 절차)** 블로그 산출물은 **좁은 폭(휴대폰 가로 360~390px) 기준으로 렌더를
  확인**한 뒤 발행한다. 뷰포트 메타(`width=device-width`)는 이미 있으나, 표/카드 단위로
  깨짐 여부를 직접 본다.
- **(적용 지점)** 산출물뿐 아니라 **생성기(generator) 소스**(예:
  `pipelines/shilla_dutyfree/build_blog_md.py` 의 표 헤더·행 렌더,
  `blog-md/assets/css/style.css` 의 `.post table`)와 CSS를 함께 고친다 —
  안 그러면 재생성 때 되돌아온다.
- **(이미지 max-width — 데스크톱에서 사진이 커지지 않게)** — CMPA-292 보드 확정 2026-06-10.
  본문 이미지는 **`max-width: min(100%, 480px)`** 로 캡한다. 읽기폭(`--read:1040px`)이
  넓은 데스크톱에서 세로 사진이 화면을 가득 채워 과도하게 커지는 것을 막기 위함이다.
  좁은 모바일에서는 `100%`라 컬럼을 넘지 않고, 데스크톱에서는 480px로 고정·가운데 정렬된다.
  이 규칙은 `.post img` 전역이라 **앞으로의 모든 블로그 글 그림에 자동 적용**된다.
  정본: `build_blog_md.py` 의 `_css()` `.post img` + `blog-md/assets/css/style.css` `.post img`
  (둘 다 같이 유지 — 재생성 때 안 되돌아오게). 값(480px)을 바꾸려면 이 두 곳을 함께 고친다.

## 블로그 필명 원칙 — 독자에게는 "보드"가 아니라 "CaskCode" (모든 agent 공통, 반드시 준수) — CMPA-198 보드 확정 2026-06-07

> **블로그(CaskCode) 글을 쓸 때, 글쓴이(저자)를 독자에게 칭할 때는 "보드"가 아니라
> 필명 "CaskCode"로 부른다.** 블로그는 독자를 향한 공개물이며, 내부 호칭인 "보드"가
> 독자 눈에 노출되면 안 된다.

- **(필명 = `CaskCode`)** 블로그 본문·소개·설명·추천 문구 등 **독자에게 보이는 글에서
  저자를 가리킬 때**는 `CaskCode`로 쓴다. 예: `보드가 직접 개발한…` → `CaskCode가 직접 개발한…`,
  `⭐보드 추천` → `⭐CaskCode 추천`.
- **(예외 — 그대로 "보드")** "보드"가 **승인 주체(거버넌스)** 를 가리키는 경우는 필명 대상이
  아니다. 예: 외부 발행 게이트 `보드 게이트 c7405e7d`, 스테이징 배너, 코드 주석의
  `CMPA-xxx 보드 …`, 이슈 코멘트·리포트 등 내부 소통(이건 [[보드 소통 원칙]] 대상).
  즉 **저자 호칭 → `CaskCode`, 승인 주체 호칭 → `보드` 유지**.
- **(적용 지점)** 산출물뿐 아니라 **생성기(generator) 소스 문자열**도 함께 고친다
  (예: `pipelines/shilla_dutyfree/build_blog_md.py`의 기둥 `desc`). 안 그러면 재생성 때 되돌아온다.

## 블로그 카피 톤 원칙 — 담백하게, 안 오글거리게 (모든 agent 공통) — CMPA-197 보드 2026-06-07

> **블로그(CaskCode) 제목·본문 등 독자에게 보이는 카피는 "담백하게(plain)" 뽑는다.
> 과장·자뻑·낯뜨거운(오글거리는) 표현을 피한다.** 사실·데이터 중심으로 차분하게 쓴다.

- **(지양)** 자기과시·감탄·드라마 톤. 예: `'위스키 가성비 회사'를 차렸습니다`, `…실록`,
  `진짜 싼 것만 골랐다`, 과한 이모지·느낌표 남발.
- **(지향)** 무엇을 했는지 담백하게. 예: `Paperclip을 이용한 위스키 데이터 분석`,
  `[신라면세] 위스키 가격 분석 및 추천`.
- **(적용)** 제목·리드·소개·추천 문구 전반 + **생성기 소스 문자열**(재생성 시 되돌아오지 않게).

## 조직·역할 원칙 — CTO는 CEO가 겸직한다 (라우팅 종착지) — CMPA-184 보드 승인 2026-06-07

> **기술 리더(CTO)는 별도 직책 없이 CEO가 겸직한다.** 라우팅 규칙의 "코드·버그·기능·
> 인프라·기술 → CTO" 종착지가 비어 있던 문제를 해소하기 위함이다.

- **CTO = CEO 겸직.** 기술 검증·아키텍처·코드리뷰·기술 표준·정본(canonical) 통합 게이트는
  **CEO 책임**으로 공식화한다. "기술 → CTO"로 라우팅되는 일은 CEO에게 모인다.
- **구현은 여전히 위임.** 코드 구현은 DataEngineer(현재 3명)에게 위임한다. CTO(=CEO)는
  *직접 코딩하는 사람이 아니라 기술을 책임지고 검증·승인하는 사람*이다 (CEO 위임 원칙과 충돌 없음).
- **재검토 트리거.** 기술 리뷰 큐가 CEO 병목이 되면 그때 전임 CTO 채용을 재검토한다
  (CMPA-161에서 파일럿이라 채용 보류했던 결정의 자연스러운 해제 조건).

## 보드 소통 원칙 — `review/` 폴더에 보고서를 쌓는다 (모든 agent 공통, 반드시 준수) — CMPA-171 보드 확정

> **보드(사람)에게 보고·검토를 요청할 때는 이슈 코멘트에만 의존하지 말고,
> 반드시 `review/` 폴더에 마크다운 보고서를 남기고 그 파일 경로를 알려준다.**
> 이유: 이슈 안에도 보고서가 있지만 보드가 자꾸 놓친다. `review/` 폴더는
> "보드가 볼 것"이 한곳에 모이는 단일 수신함이다.

1. **(작성) 보드와 소통(보고·검토·승인요청·의사결정 요청)이 필요하면
   `review/` 폴더에 `.md` 보고서를 만든다.** 이슈 코멘트로만 끝내지 않는다.
2. **(파일명) `review/<YYYY-MM-DD_HHMM>_CMPA-<번호>_<짧은제목>.md`** 형식으로 짓는다.
   **생성일+시간을 맨 앞 prefix로** 붙인다 — 보드가 파일명 정렬로 방금 만든 보고서를
   바로 찾기 위함이다. 시간은 KST 24시 `HHMM`.
   (예: `review/2026-06-07_1548_CMPA-171_보드소통-review폴더.md`)
   같은 이슈를 여러 번 보고해도 prefix가 다르므로 새 파일로 쌓인다(덮어쓰지 않음 — 누적 기록).
   **`review/` 안에 하위 폴더를 만들지 않는다** — 모든 보고서는 평평하게 두고,
   오래된 것은 보드가 수동으로 정리한다.
3. **(경로 통지) 작성 후 이슈 코멘트/보드 메시지에 그 파일의 경로를 명시한다.**
   "보고서: `review/...md`" 한 줄을 반드시 남긴다.
4. **(헤더 메타) 보고서 맨 위에 이슈번호 / 작성 agent / 작성일 / 한줄요약**을 둔다.
   (`review/README.md`의 템플릿을 사용)

자세한 규약·템플릿은 `review/README.md` 참고. 이 규칙은 모든 agent가 영구히 따른다.

## 데이터 관리 원칙 (모든 데이터·모든 agent 공통, 반드시 준수) — CMPA-156 보드 확정

> **우리의 데이터는 "한 시점의 스냅샷"이 아니라 "수집 날짜가 찍힌 누적 기록"입니다.
> 지우고 덮어쓰는 게 아니라, 갱신·갱신·갱신하며 이력을 남깁니다.**

이 리포의 **모든 데이터**(위스키 가격, 콜키지/할랄/펫 맵, 면세, 인당비용 등 예외 없이)는
다음 3원칙으로 관리합니다.

1. **(가져오기) 기존 데이터를 먼저 가져온다.** 매 실행은 백지에서 시작하지 않습니다.
   이미 적재된 정본 데이터를 불러온 뒤 그 위에서 작업합니다.
2. **(갱신) 새로 크롤링된 데이터가 있으면 해당 항목 정보를 갱신한다.**
   새 수집 결과가 있는 항목만 최신값으로 업데이트하고, 새 수집이 없는 항목은
   기존 값을 그대로 보존합니다. (전체 삭제 후 재생성이 아니라 **항목 단위 갱신**)
3. **(수집 날짜 메타데이터) 모든 데이터에는 '수집한 날짜' 메타데이터가 반드시 붙는다.**
   사용자는 이 수집 날짜를 근거로 **현재가(현재 상태)를 유추**합니다.
   따라서 수집 날짜는 데이터의 신뢰성을 판단하는 1차 신호이며, **누락·임의 변경 금지**입니다.

**실무 함의 (어기면 데이터 신뢰가 깨집니다):**
- 데이터·리포트·맵 산출물에는 항상 '수집한 날짜 / 리포트 작성일'을 노출하세요
  (참고: HTML `📅 리포트 작성일` 헤더 패턴, `pipelines/common/dated.py` 날짜 스냅샷).
- 새 크롤이 실패하거나 일부만 들어오면 **그 부분만** 갱신하고 나머지는 stale로 덮지 마세요.
  (월 롤오버 게이팅 사례: 국내 마트 월초 데이터가 부족하면 이전 달 유지 — CMPA-116)
- 수집 날짜가 오래된 데이터를 "현재가"로 단정해 보고하지 말고, 항상 "○○일 수집 기준"으로
  단서를 답니다.
- 사용자는 "이 가격이 정확히 지금 이 순간 값"이 아니라 "수집 날짜 기준값"임을 전제로
  현재가를 유추한다는 점을 코멘트·리포트에서 분명히 하세요.

  - `@whiskeypick`(https://www.youtube.com/@whiskeypick) — 2026-03~05 1차 소스.
  - `@whiskeykey`(https://www.youtube.com/@whiskeykey) — CMPA-160 신규 검증·**보드 승인
    2026-06-07**. 거의 주간 업로드로 `@whiskeypick` 6월 0행 공백을 메움.
    ⚠️ `@whiskeypick`(위스키픽)과 핸들이 비슷하나 **다른 채널**이다(혼동 주의).
- **싼디(@SSanD3)** 도 과거 시도했으나 ASR 부적합(자막이 BGM 가사·가격 0행)으로
  **수집에서 제외**합니다. 코드 `CHANNELS` 에 등록만 돼 있고 `active=False` 입니다.
- 따라서 **"싼디에서 위스키 가격을 수집한다"는 틀린 표현**입니다.
  코멘트·리포트·메모 어디서든 항상 **`@whiskeypick`·`@whiskeykey`** 로 칭하세요. (CMPA-154 → CMPA-160 보드)
- **⚠️ 위스키 수집가에는 트레이더스/코스트코 등 '지점(매장)'을 함께 기록합니다 (CMPA-446 보드
  확정 2026-06-17 — CMPA-160 정책 반전).** 보드 지시: *"트레이더스점에 따라 상품이 다른 경우가
  있다. 따라서 어느 지점인지를 같이 적어야 한다."* 과거 CMPA-160 은 "전국 매장가가 동일하니
  위치는 항상 `트레이더스`, 지점명 미표기"였으나, **지점별 재고·취급 상품이 달라** 명시적으로
  반전한다.
  - **(위치 = 기본 마트명 유지)** `위치` 컬럼은 정규화·floor 연속성을 위해 **기본 마트명**
    (`트레이더스`·`코스트코`·`우성마트`)을 그대로 유지한다. `canonical_store()` 의 전역 동작
    (지점 제거)도 바꾸지 않는다 — 데일리샷 복합 라벨 등 다른 소비자에도 쓰이기 때문.
  - **(지점은 별도 컬럼·출처에 보존)** 유튜브 OCR CSV(`{YM}_youtube_ocr.csv`)에 **`지점` 컬럼**을
    두고, 영상 제목에서 `branch_from_title()` 로 추출한 지점(`구월점`·`송도점`·`우성 식자재 마트`
    등)을 채운다. **독자에게 보이는 `출처`/리포트에도 지점을 노출**한다(예: "트레이더스 구월점 기준").
  - **(미상은 위장 금지)** 영어 제목 등 지점을 못 찾으면 **빈값**으로 둔다 — 모르는 지점을
    임의의 트레이더스 지점으로 단정하지 않는다(위치 위장 금지).
  - **(floor 는 당분간 제품키 유지)** 국내 최저가(floor)는 지점별로 분리하지 않고 제품키를
    유지하되, provenance(어느 지점에서 본 가격인지)로 `branch` 를 정규화 통합 스키마까지 전파한다.
  - **(정본 소스)** 지점 파서·표시 = `pipelines/youtube_traders/frame_ocr/ingest_ocr.py`
    (`branch_from_title`·`store_display`·`지점` 컬럼), 전파 = `scripts/normalize_dataset.py`
    `adapt_domestic`(nullable `branch`). 회귀 = `scripts/test_youtube_ocr_gate.py`.
- **⚠️ 우성식자재마트는 유튜브 OCR 수집에서 영구 제외 (CMPA-497 보드 확정 2026-06-19).** OCR 품질이
  낮아 신뢰할 수 없다. `ingest_ocr.py`의 `_EXCLUDED_STORES = {"우성식자재마트"}` 게이트가
  해당 매장 영상을 **격리(quarantine)가 아니라 전체 드롭**한다. **우성그린마트 등 다른 우성 변형은
  이번 범위 밖(별도 보드 지시가 없는 한 수집 유지).**

### 데일리샷 수집 — 면세점/해외 리스팅은 반드시 제외 (보드 확정 2026-06-13, CMPA-321) ⚠️
> **데일리샷 마켓플레이스 검색에는 면세점(신라면세 `service_type==5`)·해외 셀러가 KRW 로
> 섞여 들어온다. 면세가는 세금 0·출국/한도 조건이라 '국내 최저가'가 아니다. 데일리샷
> 수집(floor·동반 listings 모두)에서 면세/해외 리스팅은 반드시 제외한다.** 면세는 별도
> 신라면세 파이프라인이 담당한다. (과거 사건: 듀어스15년 48,864 면세가가 국내 floor 로
> 오염 — CMPA-321.)
- **판별 게이트** = `pipelines/dailyshot/crawl_dailyshot.py:is_dutyfree_listing(p)`:
  `price_usd>0` 또는 `net_price_usd>0`(1차·자가완결 신호) **또는** `service_type==5`(백스톱).
  셀러 API 가용성과 무관하게 동작한다. floor(`match_lowest`)·listings 양쪽에 적용.
- **가드레일**: 런타임 자가검증(메트릭 `dutyfree_leak`==0 확인) + 회귀 테스트
  `scripts/test_dailyshot_dutyfree_guard.py`. 수집 로직 변경 시 이 테스트를 반드시 통과시킨다.
- 이 규칙을 우회/완화하지 말 것 — 면세를 다시 들이면 국내 최저가 신호가 깨진다.

### '데일리샷 가격' = 제품 페이지(/m/item) 전국 최저 셀러가 (보드 확정 2026-06-14, CMPA-344) ⚠️
> **앞으로 "데일리샷 가격"의 의미 자체가 '제품 페이지 전국 최저가(page_price)'이다.**
> 검색 API(`/items/search`)의 `price` 는 **대표 셀러가**라 실제 전국 최저가보다 높게 나온다
> (글렌드로낙16 검색 372,000 vs 페이지 329,000/피보, 발베니19 위크오브피트 620,000 vs
> 395,000). 검색 대표가를 '데일리샷 가격'으로 노출하면 국내최저가가 체계적으로 부풀려진다.

- **정본 헬퍼** = `pipelines/shilla_dutyfree/enrich_dailyshot.py:item_page_price(tpid)`
  (제품 페이지 `__NEXT_DATA__` 의 dehydratedState 에서 최저 셀러가·셀러명 파싱).
- **`enrich_dailyshot.build_lookup(names)` 의 기본값이 `with_page=True`** — 매칭 항목마다
  페이지가를 1회 조회(tpid 1회·pace 0.7s·실패 시 검색가 폴백)해 **`ds_price` 자체를 페이지가로
  덮는다**(검색 대표가는 `ds_search_price` 로 보존). 따라서 `ds_price` 를 읽는 모든 소비자
  (`refresh_report_prices`=신라 국내최저, `detect_price_changes`=일일 가격변동,
  `build_weekly_digest`=주간 회고)가 자동으로 페이지가를 쓴다.
- **floor 리포트** `find_cheaper_than_domestic.py` 는 별도 매칭(`ds_best`)이라 main 에서
  직접 페이지가를 조회해 floor 로 쓴다(동일 방법론, CMPA-344).
- **면세/해외 제외(CMPA-321)는 페이지 파서에도 유지**(`_walk_page_price` 의 `price_usd>0` 제외).
- ⚠️ **데일리샷 크롤 정본 파이프라인**(`pipelines/dailyshot/crawl_dailyshot.py:match_lowest`)은
  아직 검색 셀러 min(페이지가 아님) → 정규화 DB floor/메인 위스키 리포트가 영향. 페이지가
  전환은 별도 이슈에서(가드 테스트 동반). 이 표식이 보일 때 그 파이프라인도 페이지가로 통일할 것.

### 위스키 '최저가' 전체 소스 그림
- **데일리샷 라이브** (`api.dailyshot.co`, `pipelines/dailyshot/`) — 국내 최저가 핵심 신호(면세 제외)
- **코스트코/트레이더스 웹** (`costcome.com`, `pipelines/costco_web/`)
- **유튜브 `@whiskeypick`·`@whiskeykey` ASR** (이마트 트레이더스 현장가, `pipelines/youtube_traders/`)
- **해외 홍콩/일본 Shopify** (`pipelines/overseas/`) — 비교 벤치마크용
- → `scripts/normalize_dataset.py` 가 정규화 id 로 합쳐 **DB floor(국내 최저가)** 산출
  → `scripts/generate_report.py` 가 리포트 생성.
- 신라면세 리포트는 별도: 국내최저 = `min(데일리샷 라이브[면세 제외], 정규화 DB floor)`
  (`pipelines/shilla_dutyfree/refresh_report_prices.py`, CMPA-141).

> **⚠️ floor(국내최저)는 '소스(매장)별 최신 관측가 중 최소값'이다 — 기간 내 단순 min() 금지
> (CMPA-496 보드 확정 2026-06-19).** 트레이더스/코스트코는 가격을 **전 지점 동일하게 오르내리므로**,
> 새 관측이 들어오면 같은 소스의 옛 관측은 **무효(superseded)**다. 기간(예: 40일) 안에서 소스
> 구분 없이 min() 을 잡으면 이미 사라진 옛 저가를 국내최저로 집어 **가격 인상을 인하처럼**
> 보이게 한다(사건: 글렌글라사 포트소이 w030 — 트레이더스 89,800(05-27)→109,800(06-09) 인데
> floor 가 89,800 으로 표기, 현재가 109,800 과 모순). **규칙: 소스별로 최신 수집일의 가격만 남기고,
> 그 최신가들 중 최소값을 floor 로 한다. 같은 소스의 과거 저가는 더 싸도 무시.** 정본 헬퍼 =
> `pipelines/common/source_floor.py:per_source_latest_floor((source,date,price)…)` →
> `(floor, source, prev)`(prev=floor 소스 직전가, ▲인상/▼인하 방향 표기용). 적용처:
> `run_ocr_collection._domestic_floor_lookup`(youtube-traders 블로그 floor),
> `analyze_attractiveness.load_domestic`(신라 DB floor). 회귀 = `scripts/test_youtube_ocr_gate.py` §11.
> ⚠️ 과거 관측 행은 **삭제하지 않는다**(누적 기록 보존·CMPA-156) — floor *선택* 로직만 최신가를 고른다.

관련 루틴: `102 · 유튜브 트레이더스 현장가 수집`(0de186c4), `위스키 가격 트래커 주간 갱신`(ee853b77),
`[All-in-one] whisky-price 통합 실행`(f66b4e58).

## 위스키 데이터 매칭·그룹핑 원칙 (모든 데이터 agent 필독) — CMPA-177 보드 확정 2026-06-07

> **위스키 데이터를 다룰 때(수집·정규화·매칭·그룹핑·리포트), '무엇이 같은 제품이고 무엇이
> 다른 제품인지'를 아래 기준으로 판단한다. 어기면 가짜 딜·오매칭이 생긴다.**

1. **(제품을 가르는 핵심 토큰 — 절대 병합 금지)** **숙성년수(`N년`), `CS`/`캐스크스트랭스`(cask
   strength), `피티드`(peated)**, 그리고 캐스크 종류(`셰리`/`버번` 등)·`에디션`. **이 토큰의 값이
   다르면 서로 다른 제품**이다. 가격 비교·동의어 병합 시 이 토큰이 한쪽에만 있거나 값이 다르면
   **다른 SKU로 취급**한다.
   - ⚠️ 실패 사례(CMPA-177): 데일리샷 검색 '제임슨'(표준)이 **'제임슨 18년'(370,000원)** 에 근접
     매칭돼 가짜 −34만원 딜이 떴다. → 숙성년수 비대칭 = 다른 제품. (가드: 데일리샷가가 마트가
     2.5배↑면 비교 숨김 — `whisky_report_tables.build_domestic`.)

2. **(용량 — 같은 제품의 변형)** `700ml`·`750ml`·`1L`·`1.75L` 등은 **같은 제품인데 용량만 다른 것**.
   다만 **현재처럼 용량별로 따로 분류해도 괜찮다**(보드 확정). 용량 토큰을 제품 구분 토큰처럼
   '다른 술'로 오해하지 말 것 — 단지 용량 변형임을 인지하고, 분류는 현행 유지.

3. **(잔세트·번들 — 노이즈, 삭제)** `전용잔`·`잔패키지`·`잔세트`·`미니어처 세트`·`메가잔`·
   `기프트/선물 세트`·`증정` 등 **번들 리스팅은 '술 한 병'이 아니라 잔값이 끼어 가격이 부풀려진
   노이즈** → **제외(삭제)**. 공통 게이트 `pipelines/common/whisky_quality.is_bundle_noise`
   (→ `is_quarantined` reason `bundle_glass_set`)가 수집·정규화·리포트에서 일괄 차단한다.

## 위스키 이름 파싱·메타데이터 규칙 (모든 데이터 agent 필독) — CMPA-735 보드 확정 2026-07-01

> **전체 규칙집 정본: `docs/whisky-naming-rulebook.md`** — 아래는 수집·분석 시 필수 요약이다.
> 이름에서 메타데이터를 뽑을 때, 또는 두 이름이 같은 제품인지 판단할 때 이 규칙을 적용한다.

### 파서 적용 순서 (이름 문자열 처리 시 반드시 이 순서로)

1. **용량 추출·제거** — `r"(\d+(?:\.\d+)?)\s*(ml|l(?![a-z]))"` → `volume_ml` 저장 후 이름에서 제거
2. **번들 노이즈 차단** — `is_bundle_noise()` 통과 못 하면 격리
3. **관사 정규화** — `The `, `더 ` 제거 (매칭용, 원문 보존)
4. **숙성년수 추출** — `r"(\d{1,2})\s*(년|years?\s*old|year\s*old|y\b)"` → `age` (정수 또는 `NAS`)
5. **CS 추출** — `r"\b(CS|C/S|cask\s*strength|캐스크스트렝스)\b"` → `is_cs` bool
6. **피티드 추출** — `r"\b(peated|heavily\s*peated|피티드|peat)\b"` → `is_peated` bool
7. **캐스크 유형 추출** — sherry/버번/port/PX/rum/double wood/triple/quarter cask/미즈나라 등 → `cask_type` 리스트
8. **색상 라벨** — 조니워커 한정: Red/Black/Green/Gold/Blue/Double Black → `label_color`
9. **에디션·배치** — `edition`, `batch`, `is_limited`
10. **나머지** → `name_ko` / `name_en` 순수 제품명

### 핵심 파싱 규칙 (자주 틀리는 것)

- ⚠️ **4자리 숫자는 `age` 아님** — 빈티지 연도(`2005`, `1990`) → `vintage_year` 별도 필드
- ⚠️ **모델 번호·도수는 `age` 아님** — `46`(메이커스 마크 46)·`101`(와일드터키 101 Proof)·`No.7` → 제품명 일부
- ⚠️ **NAS ≠ 0년** — 숙성년수 표기 없으면 `age = NAS` (NAS 제품과 년수 있는 제품 병합 금지)
- ⚠️ **500ml 미만 수집 금지** (CMPA-733) — `is_undersized_volume()` 게이트
- ⚠️ **OCR 잘린 이름은 aliases CSV로 보정** — `assets/whisky-aliases.csv` 정본 동의어·오탈자 목록
  (`pipelines/dailyshot/crawl_dailyshot.py:_load_alias_canon()` 이 `load_targets()` 에서 자동 적용)

# corkage_free — 지하철역 기준 위스키 콜키지프리 식당 탐색기 (CMPA-55, POC)

지하철역을 입력하면 그 역에서 **걸어갈 수 있는 거리**의 **콜키지프리(외부주류 반입 무료)**
식당을 찾고, 그중 **위스키 반입 신호**가 있는 곳을 우선순위로 올려준다.
가성비 위스키 ↔ 콜키지프리 식당을 잇는 송객(送客) 후크용 POC. 1차 대상 = `강남역`.

## 왜 이 소스인가 (DiningCode)

| 후보 소스 | 상태 |
|-----------|------|
| 네이버 플레이스 내부 API | ❌ 키 없이 HTTP 429 봇차단(실측) |
| 네이버 지역검색 Open API / 카카오 로컬 API | ⚠️ 클라이언트 키 발급 필요(미보유) — 프로덕션 후보 |
| **DiningCode isearch API** | ✅ **키 불필요**, "{역명} 콜키지프리" 자연어 랭킹 + 좌표/카테고리/리뷰/콜키지태그 반환 |

`POST https://im.diningcode.com/API/isearch/` 한 번으로 식당별
이름·도로명주소·전화·**lat/lng**·카테고리·점수·리뷰·키워드태그(`콜키지프리 mark=1`)를 받는다.
좌표가 있으므로 역 좌표 기준 **도보거리(haversine)** 를 계산해 반경 필터가 가능하다.

## 동작

1. `STATION_COORDS` 에서 역 앵커 좌표를 잡는다(미등록 역은 `--lat/--lng`).
2. DiningCode 를 `{역} 콜키지프리`·`{역} 콜키지무료`·`{역} 위스키 콜키지` 로 호출, `v_rid` dedup.
3. 각 식당까지 직선 도보거리·도보분 계산 → `--radius` 이내만 유지(기본 800m≈12분).
4. **위스키 신호**: 카테고리/키워드태그/리뷰의 위스키·하이볼·양주·싱글몰트·스카치·버번 언급 +
   `위스키 콜키지` 질의 매칭 여부로 가산. 신호 desc → 거리 asc 정렬.
5. `data/corkage-free/{역}_콜키지프리.{csv,md,html}` 저장(+ `_runs/` 날짜 스냅샷).
   - CSV: 전체 행(대표사진 URL·네이버지도 딥링크 포함).
   - HTML: 대표사진 썸네일 + 네이버지도 버튼 + 도보/점수/주소/전화 카드(위스키 신호 매장 강조). CEO가 바로 보기용.

## 상세 정보(CMPA-55 프로덕션 추가)
- **대표사진**: DiningCode `image`(매장 대표 이미지/메뉴컷, 300×300 webp). 전 행 보유.
- **네이버지도 링크**: `https://map.naver.com/p/search/{식당명}` — **상호만**으로 검색.
  (상호+주소를 함께 넣으면 지도가 안 뜨는 사례가 있어 CMPA-55 피드백 반영해 상호만 사용.)
- 대표 *메뉴명* 은 DiningCode 프로필이 JS 렌더라 list API/HTML 에서 안정 추출 불가 → 대표사진으로 대체(요청은 "사진 *또는* 메뉴").
- **(보류) 역 중심 개요 지도**: `mapshot.render_station_map()` 함수는 남아있으나, CEO 요청으로
  **HTML 상단 개요 지도 임베드는 제외**한다(식당별 미니지도로 충분). 함수는 향후 필요 시 재사용 가능.
- **식당별 미니 지도**(CMPA-55 피드백): 각 카드의 **음식 사진 우측 하단에 작게 오버레이**로
  역(파란 S)↔해당 식당(빨간 핀) 위치를 얹는다(92×69px, 흰 테두리). `mapshot.render_pair()` 가
  두 점이 들어오도록 줌을 자동 선택, 연결선까지 그린다. 타일은 `_TILE_CACHE` 로 프로세스 내 재사용
  (같은 역 주변이라 91장 ~7초). 이미지는 `to_data_uri()` 로 **base64 인라인** → 자가포함 HTML
  (강남역 ~0.9MB / 합정역 ~0.36MB). 첨부 HTML 을 열면 외부 파일 없이 렌더.

## 순위(추천순) 근거 (CMPA-55 피드백)
카드 좌상단 "추천 N위"는 자체 가중치가 아니라 다음 **투명한 정렬 순서**다:
① 인기·평판 **종합점수(평판점수, 0–100)** 높은 순 → ② 역에서 가까운 순.
HTML 헤더에 기준을 명시한다. (user ⭐=방문자 별점 5점, 평판점수와 별개)

## 강남역 1차 결과 (2026-05-31, 800m)

- 콜키지프리 식당 **95곳**, 그중 위스키 신호 **8곳**(세이버리·이자카야 열·먼데이블루스 등).
- 1위 세이버리: 카테고리 자체가 `와인바, 위스키` + 위스키검색 매칭(강한 신호).

## 유료 콜키지 필터 (CMPA-55 피드백 — '올드타운 홍대점' 오입력 차단)
DiningCode 크라우드 태그가 노이즈를 포함해, **유료/모호 콜키지**가 목록에 섞이는 문제를 2단계로 거른다.
- **1단계(태그)**: `콜키지프리`·`콜키지무료`(띄어쓰기·free 변형 포함) 태그가 있는 곳만 인정.
  바`콜키지`(=콜키지 토픽 태그, 유료일 수 있음)만 달린 곳은 **제외**. → '올드타운 홍대점'(태그=`콜키지`, 실제 유료) 차단.
- **2단계(리뷰 증거)**: 리뷰/키워드에서 유료·조건부 패턴(`콜키지 N원/N만`, `콜키지 비용`, `유료 콜키지`,
  `병당`, `1병당`, `2인1병`, `콜키지 받`)을 탐지하면 제외(주변에 무료/없 신호 있으면 오탐 방지로 통과).
  예: 와인만 무료·위스키 유료 같은 조건부 매장(고메램류)을 거른다.
- `--keep-suspect`: 2단계 해당 매장을 제외하지 않고 confidence 에 `⚠️유료의심(근거)` 만 표시.
- 한계: list API 는 매장당 대표 리뷰 1건만 줘서 2단계는 best-effort. 1단계(태그)가 주 방어선.
  실행 로그에 `필터: 무료태그없음 제외 N곳 · 유료의심 제외 M곳` 으로 투명 보고.

## 인당 비용 추정 (`estimate_per_person.py`, CMPA-66)

특정 콜키지프리 식당에서 한 끼 먹으면 1인당 대략 얼마인지 추정한다.

- 출처: DiningCode 프로필 페이지 JSON-LD 메뉴(정가, 라이브 GET). isearch API 에는 가격이 없다.
- 모델: 업종별 '전형적 2인 주문 바구니'를 아낌/표준/넉넉 3단계로 명시 → 합계 ÷ 인원수.
  바구니 항목은 메뉴명 부분일치로 라이브 가격을 끌어오므로 가격 변동이 자동 반영된다(가정=구성, 숫자=라이브).
- **음식값 기준.** 콜키지프리=위스키 지참이라 '술값'은 식당 매출에 거의 안 잡힘 → WK 가성비 포인트.
  '1인 1주류 필수' 매장(마담풍천)은 음료 최소금액을 1인당 가산.
- 출력: `data/corkage-free/합정역_인당비용.{csv,md}`. 현재 대상=빠넬로·짚불돈·마담풍천.
  대상 추가는 `BASKETS` 에 rid+바구니만 넣으면 됨.
- 한계: 정가 메뉴 기반 추정치(실제 객단가는 추가주문·음료로 변동). 표준 바구니 가정은 리포트에 전부 공개.
- **지도 활용(CMPA-66 board 지시):** CSV에 `rid`/`lat`/`lng` 를 실어 콜키지프리 finder CSV·지도 마커와 바로 조인 가능.
  나중에 콜키지프리 식당 지도를 만들 때 마커 팝업에 '1인 예상비용(표준)'을 띄우는 용도.

### 맵 전체 추정 (`estimate_per_person_map.py`, CMPA-66 board 확장)

콜키지프리 맵 **전 식당**(강남91+합정32=123곳) 1인 비용을 자동 추정. board: "맵의 강남·합정 식당, 그 식당의 예상 인당 비용을 근거와 함께".

- 모델 = **인당 정규화**(`estimate()`, CMPA-66 정밀화): '메뉴 한 줄 = 1인분' 가정을 버리고 각 메뉴를 **먼저 1인 가격으로 환산** 후 집계.
  - `N인 세트` → ÷N · `NNNg`(고기) → ×(1인ppg/G, 고기 ppg=180) · 수량신호 없음 → 단품 1인가.
  - 정규화된 1인가들의 25/50/75 분위 + 대분류별 사이드(`CAT_MODEL`) = 아낌/표준/넉넉.
  - **왜**: 구 모델(중앙값×인분)은 N인 세트를 1인분으로 오인+인분 곱으로 **과대계상**(청돈옥 81,500). 정규화로 **30,500** 으로 교정.
- 식당별 컬럼: `근거`(요약) + **`정규화내역`**(정규화에 쓴 메뉴별 1인 환산 내역 — board 요청, 고객 노출용).
- DiningCode AWS WAF: 빠른 연속요청 ~70+건에서 'Human Verification' 차단(≈150s 자동해제). → 1.2s 페이싱 + 160s 쿨다운 재시도 + **메뉴 캐시**(`_cache/menus.json`) resumable. 재계산은 캐시만 쓰면 재크롤 0.
- 출력: `data/corkage-free/콜키지프리_인당비용_전체.{csv,md}` (rid/lat/lng 조인키). 121/123 추정, 와인바 2곳(음식메뉴 JSON-LD 없음)만 보류.
- 한계: 대분류 사이드·ppg 는 업종 통념 상수. 정가 기준(객단가 변동 미반영). 추가 정밀화 후보 = 리뷰의 '1인 N원' 멘션 앵커.

#### HTML 카드 '주류제외 인당 비용' 배지 + 클릭 모달 (CMPA-66 board)
- 각 카드에 **🍽 주류제외 인당 N원** 배지(`_cost_badge()`). 값=`1인_표준`(음식값=주류제외). '탭하면 상세 ›' 표시.
- **카드/사진 클릭 → 상세 모달**(`_render_html` 의 `#cf-ov` + `DETAIL` JS). 모바일 hover 문제 해소(board): 모달에 큰 사진 + 비용 요약 + **정규화에 쓴 메뉴별 1인 환산 내역**(예 `4인세트 142,000원 ÷4인 = 35,500/인`, `삼겹살 150g→1인180g = 18,000/인`) + 추천위스키 + 도보/평점/주소/전화 + 지도 링크. 카드 안 `<a>` 링크 클릭은 모달 안 열리고 정상 동작.
- hover 툴팁은 제거(모바일 비호환). 상세는 `_cost_detail_html()`(툴팁/모달 공유 함수)이 생성.
- **사진 임베딩**(board): `_embed_photo()` 가 DiningCode 대표사진 URL→base64 data URI 로 HTML에 인라인(자가포함, 모바일/오프라인). 캐시 `_cache/photos.json`(재실행 재다운로드 회피, 실패 시 URL 폴백). 결과 HTML 강남 ~3.7MB / 합정 ~1.5MB(사진+미니지도 모두 내장).
- 메뉴 데이터 없는 곳(와인바 2)은 '추정 보류'(회색). 비용 CSV 없으면 배지 자동 생략.
- 재생성: `python3 find_corkage_free.py --backfill` (DiningCode 재크롤 없이 기존 CSV+비용 CSV; 사진은 캐시 사용).

## 대분류 (상위 카테고리) — 맵생성 활용 (CMPA-65)

원문 `카테고리`(DiningCode, 예 "한우, 소고기")는 너무 잘게 쪼개져 필터링이 어렵다. → 결정론적
분류기로 **상위 7종 버킷** `대분류` 컬럼을 부여한다(폴백 `기타`, 현재 두 역 모두 0건).

- 버킷: 고기 / 물고기·해산물 / 웨스턴(양식) / 일식 / 중식 / 한식·기타 / 바·주류
- 정본 분류기 = `find_corkage_free.classify_major(raw)` (`--backfill`이 사용, 라이브 재크롤 불필요).
  독립 테스트 모듈 = `category_classifier.py`(`test_category_classifier.py` 27/27 통과, 동일 택소노미).
- 재생성: `python3 find_corkage_free.py --station {역명} --radius 800` (라이브 재수집 — 안전 경로).
  ⚠️ `--backfill` 은 현재 **cross-station 손상 버그**로 사용 금지(한 역 backfill이 다른 역 정본까지 덮어씀, 2026-05-31 실측). 분류 로직은 정상, 래퍼 버그. 추적: CMPA-65 후속.
- **맵생성 핸드오프 문서: [`data/corkage-free/README.md`](../../data/corkage-free/README.md)**
  — CSV 컬럼 사전, 마커 조인키(`lat`/`lng`/`rid`), 대분류 분포, 가드레일을 한곳에 정리. 맵 만드는 에이전트는 이것부터 볼 것.

## 정직성 / 한계

- **"콜키지프리" 태그 ≠ "위스키 반입 무료" 보장.** 와인 한정·1병 무료·요일 한정 등 정책이
  매장마다 다르다(웹 교차검증으로 확인). `위스키신호`는 *간접 신호*이며 `confidence` 컬럼에
  "매장확인 필요"를 명시한다. 정책 확정은 매장 전화/예약 플랫폼 확인이 필요.
- 도보거리는 **직선 근사**(haversine). 실제 도보경로는 더 길 수 있다.
- 역 좌표는 내장 사전. 프로덕션은 공공데이터(서울교통공사 역사 좌표 CSV)로 확장 권장.

## 법무/소싱 가드레일

이 POC는 **내부 R&D(수집→측정)** 범위다. DiningCode 의 큐레이션 데이터를 **공개/상업 surface
(웹/앱/배포물)** 에 재게시하는 것은 데일리샷·싼디 건과 동일한 ToS/IP 리스크를 가지므로,
공개 전 CEO 의 법무/소싱 가드레일 승인이 필요하다. 프로덕션 단계에서는 네이버/카카오 공식
Open API(키 발급)로 1차 소스를 전환하는 것이 안전.

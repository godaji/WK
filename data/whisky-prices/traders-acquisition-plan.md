# 트레이더스 위스키 현장가 데이터 확보 방안 (CMPA-160)

> 부모: CMPA-159(데이터 수집 현황). 작성 2026-06-07, DataEngineer.
> 배경: 트레이더스 매장 현장가 전용 라이브 소스가 유튜브 `@whiskeypick`(위스키픽) **하나뿐**이라
> per-IP 429 병목으로 **2026-06엔 0행**(5월 74행) → 위스키 리포트 6월 롤오버 보류의 직접 원인.
> 우리 데이터 수집에서 **가장 취약한 지점**. → 더 안정적·지속적 확보 방안 도출 + 최소 POC 검증.

## TL;DR (권장안)

1. **🥇 권장 1순위 — `@whiskeykey`(위스키키) 채널 추가 (이미 POC 검증 + 정본 적재 완료).**
   트레이더스 전 위스키 현장가를 또렷이 읽어 주고 **거의 주간**으로 날짜 명기 영상 업로드.
   POC: 2026-06-01 영상 `dMF5i15ucJQ` ko ASR → **79행**, 날짜 자동검출, 정규화 31/79.
   → `@whiskeypick` 6월 0행 공백을 메워 `data/whisky-prices/2026-06.csv` **16→94행**.
   엔진 `CHANNELS` 에 `active=True` 로 등록 완료, `discover` 8개 트레이더스 영상 확인.
2. **🥈 권장 2순위 — ASR 수집 견고화(채널 다변화 + 페이싱 분리).**
   소스를 1개→2개로 늘리면 한 채널 429/포맷변경 시에도 다른 채널로 커버. 두 채널은
   ≥60분 페이싱 가드 때문에 **요일/실행 분리**로 운용(429 회피). PO token·재시도는 후순위.

> 핵심: **단일 소스 의존이 취약성의 근원**이므로, 가장 효과적인 처방은 "또 하나의 적합한 ASR
> 채널 추가"다. 마침 `@whiskeykey` 가 우리가 찾던 바로 그 채널이었고, 6월 공백까지 즉시 메웠다.

## 후보별 상세

### 1. ASR 수집 견고화
- 현 엔진은 이미 `.state/last_fetch.json` 로 **≥60분 per-IP 페이싱 가드**, dedup, 정규화 주석 보유.
- 추가 가능: PO token 적용(web client ko ASR), 지수backoff 재시도, 자막 캐시, 영상 우선순위 큐.
- **한계**: 단일 채널이면 그 채널이 해당 월 영상을 안 올리거나(=6월 사례) 포맷이 바뀌면 0행.
  견고화만으로는 **소스 다양성 부재**라는 근본 취약성을 못 푼다. → 후보 2와 병행해야 의미.

### 2. 추가 유튜브 채널 발굴 — `@whiskeykey`(위스키키) ★권장
- `ytsearch "트레이더스 위스키 가격"` 으로 다수 후보 발굴: 위스키키·위스커·매력적인 술·하루의위스키 등.
- **`@whiskeykey`(위스키키, `UCu1LB4T5KbkrTiy7lz-f6yQ`)** 가 최적:
  - `discover --channel whiskeykey` → **트레이더스 영상 8건**(June 1, May 26·20·11·4, Apr 28·20·13)
    = **거의 주간** 업로드(우리 기존 @whiskeypick 보다 잦음).
  - **POC 실수집** 2026-06-01 영상 `dMF5i15ucJQ` ko ASR(50KB) → 기존 parse 엔진으로 **79행**:
    신뢰도 중 69·낮음 10, 정규화 매칭 31/79, 날짜 자동검출 `2026-06-01`, 위치 `트레이더스`.
  - 진행자가 가격을 또렷이 말로 읽음(가격표 OCR 아님) → ASR 적합(싼디 부적합 사유와 대조).
- **정본 연동 완료**: `data/whisky-prices/2026-06.csv` **16→94행**(78행 append, 1행 dedup),
  실행 스냅샷 `_runs/2026-06__run2026-06-07.csv` 생성. 3원칙 준수(기존 16행 보존·항목 dedup 갱신·
  수집날짜 2026-06-01 메타).
- ⚠️ `@whiskeypick`(위스키픽)과 핸들이 비슷하나 **다른 채널**. 코드·README·CHANNELS 에 혼동주의 명시.

### 3. 트레이더스 전용 2차 웹 소스(블로그/커뮤니티/카페)
- 현 `2026-06.csv` 기존 16행이 이미 WebSearch+블로그(로라주막·CocoScan) 수동 교차확인 산출.
- **한계**: 다수 블로그가 **무날짜**(신뢰도 '낮음' 부여)라 3원칙의 '수집날짜 메타'와 충돌, WAF/구조변경
  취약. costcome.com 같은 단일 정형 소스가 트레이더스엔 부재. → **교차검증 보조용**으로만 유지.

### 4. 데일리샷 라이브 필터 강화 — ❌ 전제 오류 확인
- POC: `api.dailyshot.co/items/search/` 응답에 `service_type`·`seller_id`·smartorder thumbnail
  경로(`media/smartorder_reservation/...`)로 **채널(스마트오더) 식별은 가능**.
- 그러나 `sellers/{id}` 조회 결과 스마트오더 셀러 = **동네 주류매장**(예: `국밥쟁이 강남역점`),
  **이마트 트레이더스가 아님**. 즉 데일리샷 스마트오더는 트레이더스 매장가가 아니라 데일리샷
  제휴점 픽업가다. → **필터를 강화해도 트레이더스 현장가는 안 나온다.** 후보 4 기각.
  (기존 데일리샷 크롤은 별개로 '국내 최저가 신호'로 계속 유효 — 트레이더스 전용이 아닐 뿐.)

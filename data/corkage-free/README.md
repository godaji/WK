## 1.5 지원 역 매니페스트 (`지원역_목록.csv`) — CMPA-76

맵 제품이 **실제로 LIVE 커버하는 역**을 단일 정본 자산으로 굳힌 파일. 이전엔 "지원 역"이
코드(`STATION_COORDS`)와 `*_콜키지프리.csv` glob에만 암묵적으로 존재했다.

**역 리스트 3종 — 혼동 금지:**

| 리스트 | 위치 | 의미 |
| --- | --- | --- |
| **지원 역 (LIVE)** | `지원역_목록.csv` status=live | 맵 산출물이 실제 존재 = 강남역(91)·합정역(32) |
| 후보 역 (랭킹) | `../stations/evening-hotspot-stations.csv` | 다음에 돌릴 역 랭킹 35역 (별개 자산) |
| 좌표 사전 | 코드 `STATION_COORDS` 11역 | 호출용 좌표. 산출물 없으면 매니페스트에 status=**candidate** 로 포함 |

**status 규칙:**
- `live` — `{역명}_콜키지프리.csv` 산출물이 디스크에 실제 존재하는 역.
- `candidate` — `STATION_COORDS`에 좌표는 있으나 아직 산출물이 없는 역 (좌표 준비 완료 → `find_corkage_free.py --station {역명}` 한 줄로 LIVE 승격 가능). 이는 evening-hotspot **랭킹 35역**과는 다른 좁은 집합(= '좌표만 있고 산출물 없는' 역)이다.

**스키마** (좌표는 `lat,lng` WGS84 — 기존 CSV와 동일; CMPA-60 교훈으로 `data/stations/`와 파일·스키마 충돌 없음):
```
station, lat, lng, 식당수, 인당비용행수, status(live|candidate), 생성일, csv경로, md경로, html경로, source
```
- `식당수` = `{역명}_콜키지프리.csv` 데이터 행 수 · `인당비용행수` = `콜키지프리_인당비용_전체.csv`의 해당 역 행 수
- `생성일` = 산출물 csv 파일 mtime(날짜) — 맵 생성 시점의 결정론적 프록시

**재생성 (결정론, 재크롤 없음):**
```bash
python3 pipelines/corkage_free/build_supported_stations.py
```
새 역 맵(`{역명}_콜키지프리.csv`)을 만든 뒤 이걸 돌리면 매니페스트가 자동 갱신된다.
외부 호출 없이 디스크 상태만 스캔하므로 같은 입력 → 항상 같은 출력.

## 2. CSV 컬럼 사전 (맵 마커가 쓰는 키 ★)

```
순위, 식당명, 카테고리, 대분류★, 도로명주소, 전화, 도보거리_m, 도보_분,
콜키지태그, 위스키신호, 다코점수, user_score, 리뷰수, confidence,
대표사진, 네이버지도, 다이닝코드링크, lat★, lng★, 추천_위스키종류, 페어링근거
```

| 컬럼 | 맵생성 활용 |
| --- | --- |
| `lat` / `lng` ★ | **마커 좌표** (DiningCode 제공). haversine 도보거리의 기준. |
| `대분류` ★ | **마커 색/필터 차원.** 아래 §3 택소노미 7종. (원문 `카테고리`는 그대로 보존 — 툴팁용) |
| `다이닝코드링크` | `...profile.php?rid={rid}` — **`rid`가 식당 고유 조인키.** 인당비용 CSV·페어링과 조인. |
| `식당명` | 네이버지도 딥링크/라벨 (`네이버지도` 컬럼에 완성 URL 있음) |
| `대표사진` | 마커 팝업 썸네일 (300×300 webp URL) |
| `위스키신호` / `confidence` | 위스키 반입 신호 강조용 (간접신호 — "매장확인 필요" 명시) |
| `도보_분` / `다코점수` / `user_score` | 정렬·필터·팝업 표시 |
| `추천_위스키종류` / `페어링근거` | 팝업에 페어링 한 줄 (CMPA-61) |

> CSV는 **`utf-8-sig`(BOM)** 로 저장됨 — 파이썬에서 `open(..., encoding="utf-8-sig")` 로 읽을 것.

## 3. `대분류` 택소노미 (CMPA-65) — 상위 7종 + 폴백

DiningCode 원문 카테고리("한우, 소고기" / "이자카야, 사시미" 등)는 너무 잘게 쪼개져
필터/탐색이 어렵다. 그래서 결정론적 분류기로 **상위 7개 버킷**을 부여한다.

`고기` · `물고기·해산물` · `웨스턴(양식)` · `일식` · `중식` · `한식·기타` · `바·주류`
폴백 = `기타` (현재 두 역 모두 **0건**).

**우선순위 규칙(엣지케이스):**
- `양꼬치, 훠궈` → **중식** (중식 신호가 먼저) / `양꼬치, 양갈비` → **고기**
- `이자카야, 사시미` → **일식** (일식이 물고기보다 우선)
- 단독 `스테이크` → **웨스턴** / `한우 + 스테이크` → **고기**
- `와인바, 파스타` → **웨스턴** (바·주류는 음식 신호가 전혀 없을 때만)

### 현재 분포 (2026-05-31 backfill)

| 대분류 | 강남역(91) | 합정역(32) |
| --- | ---: | ---: |
| 고기 | 45 (49%) | 18 (56%) |
| 한식·기타 | 12 (13%) | 0 |
| 물고기·해산물 | 11 (12%) | 2 (6%) |
| 웨스턴(양식) | 8 (9%) | 5 (16%) |
| 중식 | 6 (7%) | 1 (3%) |
| 일식 | 5 (5%) | 3 (9%) |
| 바·주류 | 4 (4%) | 3 (9%) |
| **기타(미분류)** | **0 (0%)** | **0 (0%)** |

## 4. 재생성 (역별 산출물 다시 만들기)

```bash
# 권장: 라이브 재수집 → 분류까지 한 번에 (안전 경로)
python3 pipelines/corkage_free/find_corkage_free.py --station 합정역 --radius 800
python3 pipelines/corkage_free/find_corkage_free.py --station 역삼역 --radius 600
```

`대분류`는 순수 함수라 **기존 CSV만으로**도 다시 채울 수 있다(DiningCode 재호출 없음). 다만:

> ⚠️ **`--backfill` 은 현재 cross-station 손상 버그가 있어 사용 금지.** 한 역에 `--backfill` 을
> 돌리면 **다른 역의 정본 CSV·스냅샷까지 덮어써** 데이터가 섞인다(2026-05-31 실측). 분류 로직
> (`classify_major`) 자체는 정상이며, 활성 `_run_backfill`/`main` 래퍼의 중복정의·스테이션 처리
> 버그다. 수정 전까지는 위 **라이브 재수집**으로 재생성할 것. (추적: CMPA-65 후속 이슈)

분류기 API (맵생성 코드에서 직접 호출 가능):
- **`find_corkage_free.classify_major(raw: str) -> str`** — 프로덕션 분류기(`--backfill`가 사용).
- `category_classifier.category_bucket(raw)` — 동일 택소노미의 독립 모듈(테스트 27/27, `test_category_classifier.py`). 맵생성에서는 위 `classify_major`를 정본으로 쓰자.
